# Звіт до практичної роботи: кросплатформений чат «Просто Чат»

## Мета роботи

Метою роботи було розробити кросплатформений застосунок для обміну приватними повідомленнями, який можна запустити локально, у Docker Compose та в k3s/k8s інфраструктурі.

## Короткий опис застосунку

«Просто Чат» — це web-застосунок для приватного листування між користувачами. Користувач реєструється, отримує унікальний системний номер формату `PC-000-000`, додає інших користувачів у контакти за цим номером і створює приватні чати.

Повідомлення надсилаються через WebSocket, зберігаються у PostgreSQL і можуть бути отримані через REST API. Backend також публікує події у RabbitMQ, а окремий worker читає ці події через AMQP.

## Архітектура

Проєкт складається з таких частин:

| Компонент | Призначення |
|---|---|
| `frontend` | Web-інтерфейс користувача, сторінки реєстрації, входу, контактів, чатів, профілю та файлів |
| `backend` | FastAPI застосунок з REST API, WebSocket, бізнес-логікою та роботою з БД |
| `postgres` | Збереження користувачів, контактів, чатів, повідомлень і файлів |
| `rabbitmq` | AMQP broker для асинхронних подій |
| `notification_worker` | Окремий сервіс, який читає події з RabbitMQ |
| `frontend/nginx.conf` | Nginx proxy для frontend, REST API, Swagger, media та WebSocket |
| `k3s/` | Kubernetes/k3s manifests для запуску в кластері |

## Реалізація вимог завдання

| Вимога | Як реалізовано |
|---|---|
| Зберігати повідомлення в БД | Повідомлення зберігаються у PostgreSQL через SQLAlchemy model `Message`. REST endpoint `GET /api/chats/{chat_id}/messages` повертає історію чату. |
| Особисті повідомлення | Реалізовано приватні чати типу `private`. Користувач додає контакт за системним номером і створює приватний діалог. |
| Кожний приватний чат в окремому потоці | Backend має `ThreadManager`, який запускає окремий `PrivateChatThread` для приватного чату на стороні сервера. |
| Socket-технологія | Повідомлення надсилаються через WebSocket endpoint `WS /ws/chats/{chat_id}?token=<jwt>`. |
| REST API для контролерів | Реалізовано FastAPI routers: auth, users, contacts, chats, messages, files. Swagger доступний за `/docs`. |
| MVC/MVVM підхід | Backend розділений на `models`, `schemas`, `routers`, `services`, `workers`. Frontend має окремий стан застосунку і функції відображення. |
| AMQP і мікросервісна архітектура | Backend публікує події `message.created`, `file.uploaded`, `user.online` у RabbitMQ. `notification_worker` працює як окремий сервіс. |
| Docker/Podman | Є `docker-compose.yml` для локального запуску і `docker-compose.server.yml` для серверного запуску. |
| k3s/k8s | Є manifests у `k3s/`: namespace, configmap, secrets, services, deployments, statefulset, ingress. |
| Автентифікація користувача | Реалізовано JWT registration/login: `POST /api/auth/register`, `POST /api/auth/login`. |

## Робота користувача в застосунку

1. Користувач відкриває сторінку застосунку.
2. Натискає «Реєстрація» і створює акаунт.
3. Backend генерує унікальний номер користувача.
4. Інший користувач реєструється так само.
5. Користувач додає контакт за номером.
6. З контакту створюється приватний чат.
7. Повідомлення надсилаються через WebSocket і зберігаються в PostgreSQL.
8. Подія про повідомлення публікується в RabbitMQ і обробляється worker-сервісом.

## Контейнеризація

Для локальної розробки:

```bash
docker compose -p prosto-chat up --build -d
```

Для сервера:

```bash
docker compose -f docker-compose.server.yml up --build -d
```

У server compose назовні відкрито тільки frontend/nginx. API, Swagger і WebSocket проксируються через той самий домен:

- `/api/...`
- `/docs`
- `/health`
- `/ws/...`
- `/media/...`

## Розгортання в k3s/k8s

Проєкт має готові Kubernetes manifests у каталозі `k3s/`. Вони створюють namespace `prosto-chat`, запускають PostgreSQL, RabbitMQ, backend, frontend і notification worker, а також налаштовують ingress.

## Перевірка працездатності

Для автоматичної перевірки використовується `scripts/release_smoke.py`. Скрипт не використовує заготовлені дані. Він:

1. чекає `/health`;
2. реєструє двох тимчасових користувачів;
3. виконує login;
4. додає контакт за системним номером;
5. створює приватний чат;
6. відправляє REST-повідомлення;
7. перевіряє історію повідомлень у БД;
8. відправляє WebSocket-повідомлення;
9. перевіряє подію `message.created`.

Успішний результат завершується рядком:

```text
RELEASE_SMOKE_OK
```

## Висновок

У результаті роботи створено повноцінний кросплатформений чат із реєстрацією користувачів, приватними чатами, збереженням повідомлень у PostgreSQL, WebSocket-комунікацією, REST API, серверними потоками приватних чатів, AMQP-комунікацією між сервісами, Docker Compose і k3s/k8s manifests.
